# Customer Data Platform (CDP) Schema Documentation

This document provides a comprehensive structural guide to the specialized, UDM-compliant database schema designed for storing and managing omni-channel customer profile datasets.

---

## 🏢 Domain 1: Core Profile & Identity Inheritance Layer

This layer acts as the centralized identity registry, splitting generic entity tracking from specific biometric characteristics using clean object-oriented data inheritance patterns.

### 1. `CDP_PARTY_TYPE`
* **Purpose:** A rigid master lookup dictionary used to enforce categorical bounds on whether an entity record represents an individual human or a corporate workspace.
* **Columns:**
  * `PARTY_TYPE_ID` (PK): Unique string identifier designating the classification (e.g., `PERSON`, `ORGANIZATION`).
  * `DESCRIPTION`: Human-readable context explaining what structural rule properties apply to the type.

### 2. `CDP_PARTY`
* **Purpose:** The absolute root ledger and single source of truth for tracking the global existence of any actor inside the database. All identifiers, contact information vectors, and tracking points point back to this row.
* **Columns:**
  * `PARTY_ID` (PK): The master system tracking identifier (Sequential UUID or integer block generated by the application server sequence banking logic).
  * `PARTY_TYPE_ID` (FK): Relates back to `CDP_PARTY_TYPE` to guarantee structural data mapping rules can be validated.
  * `CREATED_STAMP`: Audit timestamp recording exactly when this profile first entered the system.
  * `LAST_UPDATED_STAMP`: Transaction synchronization engine index recording the most recent modification timestamp.

### 3. `CDP_PERSON`
* **Purpose:** A mandatory 1:1 behavioral sub-type extending `CDP_PARTY` to safely house consumer-specific biometric features without muddying corporate profile records.
* **Columns:**
  * `PARTY_ID` (PK/FK): Shares the identical key from `CDP_PARTY`, ensuring a cascade-delete wipe if a user asserts right-to-be-forgotten privacy rules.
  * `FIRST_NAME` / `LAST_NAME`: Extracted alphanumeric strings storing core identity names.
  * `GENDER`: Single-character administrative tracker (`M`, `F`, `O`).
  * `BIRTH_DATE`: Core chronological tracking property for age-gated compliance or marketing birthday milestone filters.

### 4. `CDP_ORGANIZATION`
* **Purpose:** A 1:1 sub-type expanding `CDP_PARTY` to represent operational merchant contexts, corporate clients, or employer configurations.
* **Columns:**
  * `PARTY_ID` (PK/FK): Strict inheritance key pointing back to the core `CDP_PARTY` parent ledger row.
  * `ORGANIZATION_NAME`: Explicit commercial legal entity title string (e.g., `XYZ Inc`).

## 🔍 Domain 2: Identity Resolution & Extended Mutable Traits

This layer handles cross-system natural lookup indices, fast tags array processing, and high-performance computed aggregate fields for fast execution loops.

### 5. `CDP_PARTY_IDENTIFICATION_TYPE`
* **Purpose:** Master category lookup catalog holding naming tags for external connected source keys (e.g., `SHOPIFY_CUST_ID`, `SHOPIFY_GRAPHQL_GID`).
* **Columns:**
  * `PARTY_IDENT_TYPE_ID` (PK): System text descriptor code defining the originating software scope.
  * `DESCRIPTION`: Full breakdown note explaining what specific platform standard format rules apply.

### 6. `CDP_PARTY_IDENTIFICATION`
* **Purpose:** A cross-reference table indexing external natural primary keys against internal `PARTY_ID` rows to eliminate identity fragmentation across channels.
* **Columns:**
  * `PARTY_ID` (PK/FK): Target master profile mapping destination.
  * `PARTY_IDENT_TYPE_ID` (PK/FK): The context source definition.
  * `ID_VALUE`: The raw string value pulled from the foreign API (e.g., `7194240516349`). **Indexed for ultra-fast lookup queries.**

### 7. `CDP_PARTY_TAG`
* **Purpose:** A normalized collection table handling arrays of unstructured text tag keywords mapped to customer profiles, entirely avoiding un-indexed, comma-separated string formatting bottlenecks.
* **Columns:**
  * `PARTY_ID` (PK/FK): The target profile identity row being tagged.
  * `TAG_VALUE` (PK): The clean text label identifier used to categorize or group users (e.g., `VIP`, `Wholesale`).

### 8. `CDP_PARTY_MARKETING_METRIC`
* **Purpose:** A performance-driven aggregate caching layout that keeps calculated consumer data points isolated from core rows to prevent operational strain during heavy segment filters.
* **Columns:**
  * `PARTY_ID` (PK/FK): Target consumer anchor reference.
  * `ORDERS_COUNT`: An integer counter cache summing historical completed order actions.
  * `TOTAL_SPENT`: High-precision decimal tracking total consumer monetary pipeline throughput.
  * `LAST_ORDER_ID` / `LAST_ORDER_NAME`: Instant text pointers to the most recent checkout payload tracking markers.
  * `LAST_CALCULATED_STAMP`: Internal cron state checkpoint showing when this data point row was last recalculated.

### 9. `CDP_PARTY_ATTRIBUTE`
* **Purpose:** A flexible catch-all key-value vault mapping variable data fields (such as Shopify's online login account `state` changes) without breaking core table designs.
* **Columns:**
  * `PARTY_ID` (PK/FK): Parent identity link.
  * `ATTR_NAME` (PK): The trait identifier (e.g., `shopify_account_state`).
  * `ATTR_VALUE`: The operational text value string (e.g., `disabled`, `invited`, `enabled`).
  * `DESCRIPTION`: Explanatory developer notation tracking which system uses this dynamic row attribute.

## 🤝 Domain 3: Comprehensive Role-to-Role Relationship Engine

This structure shifts relational links from general profiles directly to specific active roles, tracking clear contextual ties over time as defined by standard universal architecture specifications.

### 10. `CDP_ROLE_TYPE`
* **Purpose:** Functional system configuration definitions defining capacities an entity can act within (e.g., `CUSTOMER`, `EMPLOYEE`, `EMPLOYER`).
* **Columns:**
  * `ROLE_TYPE_ID` (PK): Short string value declaring the operational capacity.
  * `DESCRIPTION`: Detailed definition documentation outlining permitted behavior protocols.

### 11. `CDP_PARTY_ROLE`
* **Purpose:** An intersection matrix confirming that a particular entity is authorized to act in a given capacity starting from an explicit timeline check date.
* **Columns:**
  * `PARTY_ID` (PK/FK): Target profile mapping destination.
  * `ROLE_TYPE_ID` (PK/FK): The system capacity assigned.
  * `FROM_DATE` (PK): Timestamp marking exactly when this identity began functioning in this role.
  * `THRU_DATE`: Optional expiration timestamp nullified unless a role relationship ceases.

### 12. `CDP_PARTY_RELATIONSHIP_TYPE`
* **Purpose:** Lookup dictionary standardizing structural rules for ties between active roles (e.g., `EMPLOYMENT`, `PROFILE_MERGE`).
* **Columns:**
  * `PARTY_RELATIONSHIP_TYPE_ID` (PK): Clean descriptive link code string.
  * `DESCRIPTION`: Explanatory log text defining constraint boundaries.

### 13. `CDP_PARTY_RELATIONSHIP`
* **Purpose:** Maps a clear, time-bounded relationship directly between two separate roles, ensuring complete operational context (e.g., "Person A in the role of Employee relates to Company B in the role of Employer").
* **Columns:**
  * `PARTY_ID_FROM` / `ROLE_TYPE_ID_FROM` / `FROM_DATE_FROM` (PK/FK): Composite foreign key tracking the source actor role state.
  * `PARTY_ID_TO` / `ROLE_TYPE_ID_TO` / `FROM_DATE_TO` (PK/FK): Composite foreign key tracking the target destination actor role state.
  * `PARTY_RELATIONSHIP_TYPE_ID` (PK/FK): Defines the relationship connection type.
  * `FROM_DATE` (PK): The actual initialization timestamp of the active relationship itself.
  * `THRU_DATE`: Nullable data column indicating the termination date of the relationship.

## 📡 Domain 4: Unified Omni-Channel Engagement Vectors

This structural layer decouples raw contact strings from the customer profiles themselves, normalizing them into reusable channels categorized by distinct usage definitions.

### 14. `CDP_CONTACT_MECH_TYPE`
* **Purpose:** Master lookup categorization sorting communication formats (e.g., `POSTAL_ADDRESS`, `EMAIL_ADDRESS`, `TELECOM_NUMBER`, `ELECTRONIC_ADDRESS`).
* **Columns:**
  * `CONTACT_MECH_TYPE_ID` (PK): Unified contact formatting identifier code.
  * `DESCRIPTION`: Definition notes tracking data type behavior rules.

### 15. `CDP_CONTACT_MECHANISM`
* **Purpose:** The core parent node for all communication endpoints. It assigns a unified ID to an abstract point before specialized attributes are applied.
* **Columns:**
  * `CONTACT_MECH_ID` (PK): Master system tracking identifier for the contact method.
  * `CONTACT_MECH_TYPE_ID` (FK): Links to the master lookup to ensure clean type validation.

### 16. `CDP_EMAIL_ADDRESS`
* **Purpose:** Specialized 1:1 table extension of `CDP_CONTACT_MECHANISM` housing unique email communication string paths.
* **Columns:**
  * `CONTACT_MECH_ID` (PK/FK): Strict inheritance key mapping back to the generalized mechanism.
  * `EMAIL_ADDRESS`: The raw SMTP routing text string. **Enforced with a Unique Index constraint.**
  * `IS_VERIFIED`: Boolean flag checking if direct verification loops have confirmed email ownership.

### 17. `CDP_TELECOM_NUMBER`
* **Purpose:** Specialized 1:1 table extension of `CDP_CONTACT_MECHANISM` storing clean, parsed telephone components for automated dialer or SMS text engines.
* **Columns:**
  * `CONTACT_MECH_ID` (PK/FK): Inheritance key link.
  * `COUNTRY_CODE` / `AREA_CODE`: Parsed dialing parameters.
  * `CONTACT_NUMBER`: The baseline target telephony digit sequence string.

### 18. `CDP_ELECTRONIC_ADDRESS`
* **Purpose:** Specialized 1:1 table extension of `CDP_CONTACT_MECHANISM` tracking social handles and digital communication routing details.
* **Columns:**
  * `CONTACT_MECH_ID` (PK/FK): Functional routing mechanism mapping link.
  * `SOCIAL_HANDLE`: The target channel username handle text (e.g., `@customer_handle`).
  * `PLATFORM_NAME`: Target network identification name (e.g., `INSTAGRAM`, `TIKTOK`).

### 19. `CDP_POSTAL_ADDRESS`
* **Purpose:** Specialized 1:1 table extension of `CDP_CONTACT_MECHANISM` capturing structural delivery properties for accurate physical parcel routing.
* **Columns:**
  * `CONTACT_MECH_ID` (PK/FK): Inheritance key routing link.
  * `TO_NAME`: Attention routing string (Allows seamless placement of corporate references like `c/o XYZ Inc` without creating extra parent entries).
  * `ADDRESS1` / `ADDRESS2`: Primary location description text blocks.
  * `CITY`: Target geographic city destination string.
  * `STATE_PROVINCE_GEO_ID`: Geographic region reference index parameter (e.g., standard ISO alphanumeric codes like `NY`).
  * `COUNTRY_GEO_ID`: Master standard country classification code (e.g., `US`, `IN`).
  * `POSTAL_CODE`: Local zip code delivery route sequence.

### 20. `CDP_CONTACT_MECH_PURPOSE_TYPE`
* **Purpose:** Defines the functional business intent for a communication path (e.g., `SHIPPING_LOCATION`, `BILLING_LOCATION`, `DEFAULT_LOCATION`).
* **Columns:**
  * `CONTACT_MECH_PURPOSE_TYPE_ID` (PK): Naming key defining the intended use case.
  * `DESCRIPTION`: Detailed operational notes outlining when systems should prioritize this contact path.

### 21. `CDP_PARTY_CONTACT_MECH`
* **Purpose:** A date-bounded configuration matrix link connecting profile identities to contact points under explicit use parameters, allowing one profile to cleanly hold multiple active addresses.
* **Columns:**
  * `PARTY_ID` (PK/FK): Target identity row.
  * `CONTACT_MECH_ID` (PK/FK): Target contact endpoint.
  * `CONTACT_MECH_PURPOSE_TYPE_ID` (PK/FK): Defines the usage context.
  * `FROM_DATE` (PK): Timestamp marking when this record became the active choice for this purpose.
  * `THRU_DATE`: Nullable field indicating when an address was deprecated or swapped out.

## 🎯 Domain 5: Marketing Consent & Behavioral Audit Layers

This analytics layer tracks active compliance preferences and time-series customer touchpoint actions used to power real-time cohort segmentation engines.

### 22. `CDP_PARTY_PREFERENCE_TYPE`
* **Purpose:** Dictionary managing opt-in tracks compliance parameters (e.g., `EMAIL_MARKETING`, `SMS_MARKETING`).
* **Columns:**
  * `PARTY_PREF_TYPE_ID` (PK): Clean text identifier code string.
  * `DESCRIPTION`: Detailed legal or functional context outline notes.

### 23. `CDP_PARTY_PREFERENCE`
* **Purpose:** Time-bounded preferences ledger tracking the current active opt-in compliance state of a profile across multiple communication tracks.
* **Columns:**
  * `PARTY_ID` (PK/FK): Target profile identity anchor.
  * `PARTY_PREF_TYPE_ID` (PK/FK): Target tracking channel parameters.
  * `PREFERENCE_VALUE`: Active selection status state string value (e.g., `subscribed`, `unsubscribed`).
  * `FROM_DATE` (PK): Exact date and time stamp logging when a compliance consent preference update occurred.
  * `THRU_DATE`: Nullable field deactivated when newer consent records are stamped.

### 24. `CDP_BEHAVIORAL_EVENT_TYPE`
* **Purpose:** Lookup standard tracking keys describing raw interaction actions (e.g., `page_viewed`, `product_purchased`).
* **Columns:**
  * `EVENT_TYPE_ID` (PK): Operational tracking keyword string.
  * `DESCRIPTION`: Definition notes outlining what customer touchpoint properties apply.

### 25. `CDP_BEHAVIORAL_EVENT`
* **Purpose:** An immutable, time-series logging ledger capturing real-time user touchpoints, clickstream parameters, and dynamic transactional data payloads.
* **Columns:**
  * `EVENT_ID` (PK): Unique generated tracking reference key.
  * `PARTY_ID` (FK): The consumer profile who performed the action. **Composite index included for ultra-fast timeline queries.**
  * `EVENT_TYPE_ID` (FK): Action classification link.
  * `EVENT_DATE`: Timestamp capturing precisely when the action occurred.
  * `METRIC_VALUE`: Optional decimal field for quick numeric processing (e.g., cart value thresholds).
  * `PAYLOAD_JSON`: Unstructured JSON block storing raw contextual metadata directly from API webhooks for analytical pipelines.

### 26. `CDP_SEGMENT`
* **Purpose:** The campaign rules catalog housing filter definitions and target grouping logic criteria.
* **Columns:**
  * `SEGMENT_ID` (PK): Unique segment identity tracker code.
  * `SEGMENT_NAME`: Descriptive title given to the target audience (e.g., `High Value Inactive Shoppers`).
  * `RULE_LOGIC_EXPRESSION`: Heavy text string storing raw script conditions evaluated against customer logs.

### 27. `CDP_SEGMENT_MEMBERSHIP`
* **Purpose:** A historical roster tracking active customer inclusion timelines within targeted marketing segments.
* **Columns:**
  * `SEGMENT_ID` (PK/FK): Target audience campaign tracker node.
  * `PARTY_ID` (PK/FK): The consumer profile captured by the filter rules.
  * `EVALUATED_DATE` (PK): Timestamp tracking exactly when the classification evaluation engine ran.
  * `IS_ACTIVE`: Boolean flag indicating if the customer currently qualifies for this audience group.
